1️⃣ Начать с бекенда

Почему:

Backend — это «ядро» системы. Все функции, включая регистрацию, RBAC и аудит, будут использоваться и фронтендом, и будущими модулями.

Если сразу сделаешь фронтенд, а бекенд ещё «сырой», фронтенд будет зависеть от моков и придётся потом переписывать интеграцию.

Базовые API — регистрация, логин, роли — можно протестировать и утвердить до дизайна фронтенда.

Что делать на первом этапе бекенда:

Настроить проект (TS + Node + Express/Fastify)

Сделать модуль users + auth

Сделать RBAC и middleware для проверки прав

Настроить Audit layer (лог действий)

Тестировать API через Postman / Insomnia

Результат: рабочий backend, готовый принимать запросы с фронтенда, полностью протестирован и безопасен.

2️⃣ После базового backend — frontend

Почему:

Теперь фронтенд может работать с настоящим API, а не с моками.

Фокус можно сделать на UI/UX, формах регистрации, логине, просмотре ролей и истории действий.

Фронтенд строится поверх стабильного ядра и легко интегрируется с будущими бизнес-модулями.

Что делать на фронтенде:

Настроить проект (React + TS + Vite)

Настроить роутинг, формы и валидацию (Zod + React Hook Form)

Сделать страницы: логин, регистрация, профиль, роли

Настроить state management (Zustand / TanStack Query)

Подключить backend API для взаимодействия

Результат: UI, полностью связанный с backend, готовый для расширения под бизнес-функции.

3️⃣ После этого — параллельная разработка

Когда ядро готово и фронтенд умеет работать с API:

Можно параллельно добавлять новые модули backend: Rack, Battery, Export

Фронтенд развивает интерфейсы для этих функций

Используется та же структура use-case → domain → repository → controller

✅ Итог

Сначала backend: ядро, auth, RBAC, аудит

Потом frontend: подключение к API, формы, просмотр данных

Дальше параллельно: бизнес-фичи и интерфейсы под них

Если хочешь, я могу сделать пошаговую дорожную карту с конкретными задачами по бекенду и фронтенду на 4–5 спринтов, чтобы сразу видеть что и когда делать, чтобы MVP с регистрацией и базовыми инструментами заработал максимально быстро.

🗂️ Дорожная карта разработки WorkNest / CoreSpace
Sprint 1 — Backend: фундамент ядра

Цель: создать базовый backend с безопасным доступом и ядром для будущих модулей

Задачи:

Инициализация проекта (Node + TypeScript + Express/Fastify)

Настройка линтинга, тестов, конфигов (ENV, DB, JWT)

Модуль users:

Entity: User

Repository: UserRepository

Use-cases: registerUser, loginUser, resetPassword, verifyEmail

RBAC:

Entity: Role, Permission

Use-case: checkPermission

Middleware проверки прав на маршрутах

Audit:

Entity: AuditEvent

Use-case: logAction

Middleware логирования действий

Тестирование: unit и integration тесты для всех use-cases и middleware

Результат: backend с регистрацией, логином, ролями и аудитом, полностью готовый к подключению фронтенда

Sprint 2 — Frontend: базовый интерфейс

Цель: создать UI для регистрации, логина и работы с пользователями и ролями

Задачи:

Инициализация React + TypeScript + Vite

Настройка маршрутов (React Router)

Формы: регистрация, логин, сброс пароля, профиль

Валидация форм: React Hook Form + Zod

State management: Zustand + TanStack Query для работы с API

Подключение backend API:

Регистрация и логин

Просмотр профиля и роли

Проверка прав доступа

Тестирование компонентов и страниц

Результат: фронтенд, полностью связанный с backend, с авторизацией и просмотром ролей

Sprint 3 — Backend: первая бизнес-фича (Rack)

Цель: создать рабочий модуль расчёта, используя ядро безопасности и аудит

Задачи:

Модуль rack:

Entity: RackComponent

Repository: RackRepository

Domain: calculateRack

Use-case: calculateRackUseCase

Контроллер и маршруты: /rack/calculate

Интеграция с RBAC (только авторизованные пользователи)

Логирование действий через Audit middleware

Тестирование: unit и integration тесты

Результат: первый бизнес-инструмент готов и защищён ядром приложения

Sprint 4 — Frontend: интерфейс Rack

Цель: сделать удобный UI для расчёта и сохранения стеллажей

Задачи:

Страница расчёта стеллажей

Подключение API расчёта

Отображение результатов, стоимости, выбранных компонентов

Возможность сохранять конфигурации (RackSet)

Логирование действий и уведомления (UI feedback)

Тестирование компонентов и взаимодействий

Результат: рабочий UI для первой бизнес-фичи, полностью интегрированный с backend

Sprint 5 — Backend + Frontend: расширение функционала

Цель: добавить батареи, экспорт, ревизии и фильтры

Задачи:

Модуль battery:

Domain: calculateBatteryFit

Use-case: calculateBatteryUseCase

Интеграция с Rack

Экспорт: Excel / PDF

Ревизии и soft delete / restore

Фронтенд:

Страницы батарей и экспорта

Управление версиями и восстановление

Фильтры и пагинация

Тестирование: unit + integration

Результат: расширенный workspace с первой полноценной бизнес-логикой и возможностью экспорта

✅ Итоговая стратегия

Backend сначала: создаём ядро (users, auth, RBAC, audit)

Frontend потом: подключаем к реальному API

Параллельно расширяем: новые бизнес-модули (rack, battery, export)

Каждая новая фича использует уже готовое ядро, тесты и audit
